# Auth0 JWT Validation — Refactored by Claude Code

This repo (`master` branch) contains the refactored version of the Flask app produced in the tutorial [Next-Gen DevEx: Integrating Claude Code with Auth0 APIs](<auth0-tutorial-link>).

It was generated by Claude Code using the Auth0 MCP Server. The application was provisioned in the Auth0 tenant and the validation code was written in the same session, keeping the two in sync from the start. See the [`suboptimal-implementation`](https://github.com/manishh/Claude-Code-With-Auth0-MCP/tree/suboptimal-implementation) branch for the suboptimal starting point (naive Flask app) this was refactored from.

---

## What Changed and Why

### `app.py`

**`OIDCProvider` class**

Discovers the issuer and JWKS URI dynamically from the tenant's OIDC discovery document (`/.well-known/openid-configuration`) rather than hardcoding them. The JWKS is cached with a configurable TTL (`JWKS_CACHE_TTL_SECONDS`, default 3600s) and refreshed automatically on expiry. On an unknown `kid`, it performs a single forced refresh before giving up. This handles key rotation without requiring a restart.

**`kid` matching**

The original code always validated tokens against the first key in the JWKS regardless of which key signed the token. The refactored code extracts the `kid` from the token header and selects the matching key explicitly. This was the critical finding from the audit: the tenant was actively publishing two rotating RS256 keys, meaning roughly half of all tokens were failing silently.

**Typed exception handling**

The bare `except` that swallowed all validation errors identically was replaced with specific handlers for `ExpiredSignatureError`, `JWTClaimsError`, and `JWTError`. Each maps to a stable, non-specific error code returned to the client (`token_expired`, `invalid_claims`, `invalid_token`). Full error causes are logged server-side only, so validation internals are never leaked to callers.

**Scope enforcement**
A `require_scope` decorator was added to enforce API-level permissions on protected routes. The required scope (`API_REQUIRED_SCOPE`, default `read:data`) is checked against the `scope` claim in the validated token, returning `403 insufficient_scope` on failure.

**Clock skew tolerance**

A configurable leeway (`AUTH0_LEEWAY_SECONDS`, default 60s) was added to `jwt.decode` to handle minor clock drift between the token issuer and the resource server.

**Fail-fast configuration validation**

The app raises a `RuntimeError` at startup if `AUTH0_DOMAIN` or `AUTH0_AUDIENCE` are missing or still set to placeholder values, rather than producing confusing auth errors at request time.

### `get_token.py`

A separate dev utility for minting access tokens using the client credentials flow. It is intentionally kept out of `app.py` because a resource server has no business holding a client secret. Use it to generate Bearer tokens for local testing without going through the Auth0 dashboard each time.

## Setup

**1. Install dependencies**

```bash
pip install -r requirements.txt
```

**2. Populate `.env`**

```
AUTH0_DOMAIN=your-tenant.us.auth0.com
AUTH0_AUDIENCE=https://your-api-identifier
AUTH0_CLIENT_ID=your-m2m-client-id
AUTH0_CLIENT_SECRET=your-m2m-client-secret
```

`AUTH0_CLIENT_ID` and `AUTH0_CLIENT_SECRET` are only needed for `get_token.py`. `app.py` (the resource server) does not use them.

**3. Run the app**

```bash
python app.py
```

**4. Mint a test token and hit the protected endpoint**

```bash
TOKEN=$(python get_token.py)
curl -H "Authorization: Bearer $TOKEN" http://localhost:5000/api/protected
```

Expected response:

```json
{"message": "Access granted to a protected resource", "sub": "your-client-id@clients"}
```

## Configuration Reference

| Variable | Default | Description |
|---|---|---|
| `AUTH0_DOMAIN` | required | Your Auth0 tenant domain |
| `AUTH0_AUDIENCE` | required | API identifier (resource server) |
| `AUTH0_CLIENT_ID` | required | Your Auth0 client ID |
| `AUTH0_CLIENT_SECRET` | required | Your Auth0 client secret |
| `AUTH0_LEEWAY_SECONDS` | `60` | Clock skew tolerance for token expiry |
| `API_REQUIRED_SCOPE` | `read:data` | Scope enforced on `/api/protected` |
| `AUTH0_HTTP_TIMEOUT` | `5` | Timeout in seconds for Auth0 HTTP calls |
| `JWKS_CACHE_TTL_SECONDS` | `3600` | How long to cache the JWKS before refreshing |
| `FLASK_DEBUG` | `false` | Enable Flask debug mode (never set in production) |

## A Note on Dependencies

The refactored code uses `python-jose` correctly with proper `kid` matching, JWKS caching, and typed error handling. For production use, consider migrating to PyJWT's `PyJWKClient`, which subsumes the `OIDCProvider` plumbing into a well-maintained abstraction. That migration is a dependency change beyond the scope of this tutorial.
